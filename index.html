The Language

Introducing Sanguis

Sanguis is a self-hosting programming language. Using 10 keywords as the foundation, coding is as simple as doing. No abstract math. No boilerplate. No extreme syntax or hidden rules. You structure programs out of bodies that hold state, fire when conditions are met, and emit to each other. It compiles to a .vana genome file that a runtime executes. The compiler and interpreter are themselves written in Sanguis to demonstrate portability at every level.

Today, Sanguis compiles to standalone .exe binaries on normal Windows hardware. Support for linux and ubuntu based users will be released alongside the public sourcecode in the respective VolkanaDEV issued SanguisSDKs You do not need a server farm, a GPU cluster, or a cloud account. We provide the support, you create something real. The parallel capabilities needed are already inside your laptop or home computer, and serve as the foundation of the future of Sanguis and VANA itself.

Structured Like Biology

Programs are bodies. Each body holds its own state and connects to the next. No classes, no inheritance, no object hierarchies. Just bodies that breathe.

Unbounded Data Flow

Data moves through the entire fabric at once. No single-file line. No bottleneck. The GPU sparse skip method hops over empty tracks and draws less power than the baseline.

Everyday Hardware Native

You do not need a data center. Sanguis compiles to .exe binaries that run on the laptop you already own. The installer is 25 MB and needs nothing else installed.

The Engine

What is a VANA?

VANA stands for Voltage Activated Neural Architecture. The name describes the idea: computation activated through voltage pathways, structured like a neural system rather than a sequential pipeline.

In practice, VANA is the runtime engine that executes Sanguis programs. When you write Sanguis code, the compiler produces a .vana file. VANA reads that file and runs it directly on your GPU hardware, relaxing the requirements your CPU needs to execute complex real-time computing commands.

This allows for programming speeds highly sought after for years via the GPU-as-processor principal seen implemented in Sanguis.

The architecture matters because it changes what the hardware is doing rather than reinventing the wheel. Instead of forcing data through a single processing line, or proposing a linear upgrade to hardware like a new CPU or RAM, VANA routes it across many paths simultaneously on GPU hardware.

Treating it as the native platform, it skips empty regions instead of processing them on multi-core arrarys that support paralell processing, such as CUDA by NVIDIA or Intel's Xe3 . It treats the silicon it's aimed at as the platform itself, letting the CPU take a load off for once.

A new CPU core asks how to make the same thing faster. VANA asks what if the hardware did something different. Today the engine runs on CUDA cores and Vulkan compute pipelines. Tomorrow it can run on custom firmware on a dedicated board. Eventually it can run on a SoC where the language and the chip share the same structure.

The hardware may change underneath, but the language, and support for older hardware, does not. VANA is not tied to any preset configuration in silicon, firmware, or software. It is meant to serve as the bridge between Sanguis coding langauge and native hardware support for GPU-based compiled porgrams, until custom SoC allows full native integration.

Sparse Skip Routing

VANA maps data across the GPU and skips empty regions entirely. Instead of processing every cell in a grid, it jumps over the ones that hold nothing. Less power, less time, same result.

Hardware-Native Execution

VANA does not emulate or interpret. It compiles .vana files into operations the hardware already understands. The GPU compute pipelines execute the work directly. No translation layer, no overhead.

From Code to Silicon

The Sanguis-to-VANA pipeline targets consumer GPUs today. The same architecture scales to custom silicon. When the hardware changes, the engine adapts. The language stays the same.
