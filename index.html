The Language and the Engine

Introducing Sanguis

Sanguis is a self-hosting programming language. Ten keywords: body, hold, when, but, emit, stream, to, route, through, repeat. No abstract math. No boilerplate. You structure programs out of bodies that hold state, fire when conditions are met, and emit to each other. It compiles to a .vana genome that a runtime executes. The compiler and interpreter are themselves written in Sanguis. The language builds itself.

This runs today. Sanguis compiles to standalone .exe binaries on normal Windows hardware. You do not need a server farm, a GPU cluster, or a cloud account. The parallel capabilities already inside your laptop are enough.

Structured Like Biology

Programs are bodies. Each body holds its own state and connects to the next. No classes, no inheritance, no object hierarchies. Just bodies that breathe.

Unbounded Data Flow

Data moves through the entire fabric at once. No single-file line. No bottleneck. The GPU sparse skip method hops over empty tracks and draws less power than the baseline.

Everyday Hardware Native

You do not need a data center. Sanguis compiles to .exe binaries that run on the laptop you already own. The installer is 25 MB and needs nothing else installed.

The Engine

What is a VANA?

VANA stands for Voltage Activated Neural Architecture. The name describes the idea: computation activated through voltage pathways, structured like a neural system rather than a sequential pipeline. In practice, VANA is the runtime engine that executes Sanguis programs. When you write Sanguis code, the compiler produces a .vana file. VANA reads that file and runs it on real hardware.

The architecture matters because it changes what the hardware is doing. Instead of forcing data through a single processing line, VANA routes it across many paths simultaneously. It skips empty regions instead of processing them. It treats the silicon underneath as the computation itself, not a surface for interpretation. What that means in practice depends on what you build with it.

A new CPU core asks how to make the same thing faster. VANA asks what if the hardware did something different. Today the engine runs on CUDA cores and Vulkan compute pipelines. Tomorrow it can run on custom firmware on a dedicated board. Eventually it can run on a SoC where the language and the chip share the same structure. The hardware changes underneath. The language does not. VANA is not tied to any one silicon. It is the layer that adapts to whatever the hardware can do.

Sparse Skip Routing

VANA maps data across the GPU and skips empty regions entirely. Instead of processing every cell in a grid, it jumps over the ones that hold nothing. Less power, less time, same result.

Hardware-Native Execution

VANA does not emulate or interpret. It compiles .vana files into operations the hardware already understands. The GPU compute pipelines execute the work directly. No translation layer, no overhead.

From Code to Silicon

The Sanguis-to-VANA pipeline targets consumer GPUs today. The same architecture scales to custom silicon. When the hardware changes, the engine adapts. The language stays the same.

Where to Next

Custom Silicon

Today, VANA drives prototypes using the GPU hardware already on your desk. That is the starting point, not the destination. The end goal is custom silicon designed around the Sanguis architecture from the ground up. Chips where the language and the hardware share the same structure. No translation, no drivers, no abstraction layers between code and silicon. The computation becomes the chip itself.
