Hello, VANA.

Voltage Activated Neural Architecture. The runtime engine that executes Sanguis programs on real hardware. Designed to evolve as we do, every step of the way.

The NameWhat VANA Means

VANA stands for Voltage Activated Neural Architecture. It describes how the engine thinks about computation: voltage pathways activating across a structure that resembles a neural system, not a sequential pipeline.

A traditional CPU processes instructions one at a time along a single line. It does this very fast, but the model is the same regardless of how many cores you stack. VANA changes the model. Data activates across many paths at once. The hardware does the work simultaneously, not sequentially.

How It WorksThe Runtime Engine

When you write Sanguis code, the compiler produces a .vana file. This file is not an executable in the traditional sense. It is a structured description of how data should flow through the hardware. VANA reads that file and maps it directly onto the compute surface available.

Today, that surface is the GPU. VANA uses Vulkan compute pipelines to route data across thousands of parallel cores simultaneously. It does not emulate a CPU on the GPU. It does not translate instructions. It treats the GPU as the processor and maps the computation to what the silicon can actually do.

The key mechanism is sparse skip routing. When data passes through a grid, most cells are empty. A traditional approach processes every cell. VANA skips the empty ones. It jumps over regions that hold nothing and only activates the paths that matter. Less power. Less time. Same result.

Sparse Skip Routing

VANA maps data across the GPU and skips empty regions entirely. Instead of processing every cell in a grid, it jumps over the ones that hold nothing. Less power, less time, same result.

Hardware-Native Execution

VANA does not emulate or interpret. It compiles .vana files into operations the hardware already understands. The GPU compute pipelines execute the work directly. No translation layer, no overhead.

From Code to Silicon

The Sanguis-to-VANA pipeline targets consumer GPUs today. The same architecture scales to custom silicon. When the hardware changes, the engine adapts. The language stays the same.

Beyond the GPUHardware Independence

A new CPU core asks how to make the same thing faster. VANA asks what if the hardware did something different. Today the engine runs on CUDA cores and Vulkan compute pipelines. Tomorrow it can run on custom firmware on a dedicated board. Eventually it can run on a SoC where the language and the chip share the same structure.

The hardware changes underneath. The language does not. VANA is not tied to any one silicon. It is the layer that adapts to whatever the hardware can do. You are not making the same thing faster. You are changing what the hardware is for.

Where It GoesConnectome Architectures

The long-term vision for VANA is not just a faster runtime. It is a fundamentally different way for hardware to compute. The architecture maps naturally to connectome-style systems, where computation flows through interconnected pathways the way signals flow through biological neural tissue.

Working prototypes have already demonstrated this partially. The sparse skip routing, the parallel path activation, the region-based data flow. These are not theoretical. They are running today on consumer hardware. The prototypes are incomplete, but the structural foundation is proven. The engine routes data the way a connectome would, even on silicon that was not designed for it.

Current Prototype Status

The VANA runtime executes Sanguis programs on consumer GPUs today. Sparse skip routing, region-tiled processing, and parallel pipeline execution are all functional. The Vulkan compute backend runs five distinct pipelines: base, payload, shared, region, and sparse. Benchmark results on the Benchmarks page show real performance data from this hardware.

On custom silicon, the abstraction layers disappear. The language and the hardware share the same structure. Data does not pass through drivers or translation layers. It activates directly through the pathways etched into the chip. The computation becomes the hardware. That is the destination. The prototypes are the first step.

The PathTimeline

VANA is not a future concept. It is a working engine with a proven architecture.

Today: Consumer GPU Prototypes

VANA runs on consumer NVIDIA and Vulkan-compatible GPUs. The Sanguis compiler produces .vana files. The engine executes them through five Vulkan compute pipelines. Sparse skip routing, region-tiled processing, and parallel data flow are all functional. Benchmarks show real numbers on real hardware.

Next: Custom Firmware on Dedicated Boards

The same architecture maps to custom firmware running on dedicated hardware boards. No GPU driver stack. No operating system abstraction. The engine talks directly to the silicon. The .vana file becomes firmware, not a runtime payload. Faster, lower latency, lower power.

Then: SoC for Native Computation

The destination is a System on Chip where the Sanguis language and the VANA architecture are etched into the same silicon. The computation is the chip. No layers between code and hardware. The connectome structure maps directly to the physical pathways in the silicon. This is where the architecture stops being a runtime and becomes the hardware itself.
